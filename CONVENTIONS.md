# 프로젝트 규약 (Conventions)

> 폴더명·경로·도구는 작업자가 정한다. 이 문서는 **구조가 바뀌어도 유지되는 개념·규칙**만 담는다.

---

## 1. 아키텍처

> **한 시스템이 여러 "축"에서 서로 다른 아키텍처 모자를 쓴다.** 하나의 이름으로 못 줄이는 게 정상이며,
> 각 축은 서로 다른 문제(배포 / 조정 / 경계 / 도메인 / 객체)를 해결한다. 아래 5축을 따로 이해하면 된다.

| 축 | 채택 | 한 줄 |
|---|---|---|
| 배포 | 모듈러 모놀리스 | 하나의 프로세스, 안에서 도메인별로 모듈·권한 격리 |
| **에이전트 조정** | **허브-스포크 (스타 토폴로지)** | 중앙 허브가 모든 에이전트를 중재, 에이전트끼리 직접 연결 ❌ |
| 의존 경계 | 헥사고날(라이트) | 코어를 외부(DB·LLM·HTTP)로부터 어댑터로 격리 |
| 도메인 | DDD(부분) | 도메인별 패키징 + 애그리거트 경계 규율 1개 |
| 객체 | SOLID | 포트/주입/모듈분리로 SRP·DIP·ISP 유지 |

### 1-1. 배포 — 모듈러 모놀리스
- 물리적으로 **단일 프로세스/단일 배포 단위**다. MSA(마이크로서비스)로 쪼개지 않는다 — 현 규모엔 운영 비용만 늘어 과설계.
- 대신 **안에서 도메인별로 모듈·권한을 격리**해 소유권과 보안 경계만 챙긴다. ("한 박스 안의 여러 방.")

### 1-2. 에이전트 조정 — 허브-스포크 = ★ 스타 토폴로지
AI 엔진의 **에이전트들은 서로 직접 말하지 않는다.** 모든 호출·데이터 전달은 **중앙 허브(커널: 오케스트레이터+레지스트리)** 를 거친다. 네트워크로 치면 **스타(성형) 토폴로지** — 중심 노드 하나에 나머지가 방사형으로 붙고, 단말끼리는 직접 간선이 없다.

```
                      [ 채팅 ]
                         |
   [ 비용(예정) ] ── ( 커널 = 허브 ) ── [ 시뮬레이터 ] ← 큰 에이전트 1개
                                            └─ 광고분석 (내부 단계)
                                            └─ 페르소나 생성 → 반응 → 집계 → … → 리포트

   ● 허브 = 오케스트레이터(흐름 조율) + 레지스트리(에이전트 등록/탐색)
   ● 스포크 = 각 에이전트(시뮬레이터·채팅·비용). 허브를 통해서만 연결됨
   ● 광고분석은 독립 스포크가 아니라 **시뮬레이터 에이전트 안의 한 단계**
   ✗ 스포크 ↔ 스포크 직접 호출 금지
```

> **시뮬레이터는 하나의 큰 에이전트**이고, 광고분석은 그 안에 속한 개념(파이프라인의 첫 단계)이다 — 허브에 직접 붙는 별도 에이전트가 아니다.

**왜 스타인가 (mesh 대비 이점)**
- **결합도 N→1**: 에이전트끼리 그물(mesh)로 엮으면 연결이 N×N로 폭발하고 서로를 알아야 한다. 스타는 각자 **허브만** 알면 된다 → 에이전트는 서로를 모른 채 독립 개발·교체 가능.
- **단일 조정 지점**: 흐름 순서·재시도·진행률(SSE)·추적을 허브 한 곳에서 통제. 단말에 흩어지지 않는다.
- **추가가 쉽다**: 새 에이전트는 **허브(레지스트리)에 등록**만 하면 끝. 다른 에이전트는 수정 0.
- 트레이드오프: 허브가 단일 장애점/병목이 될 수 있음 → 허브는 **얇게(조율만)**, 무거운 일은 스포크가 한다.

> 같은 "중심 하나" 패턴이 배포에도 반복된다: **엣지 게이트웨이 1개**가 외부 진입의 허브(§3). 즉 *조정의 허브=커널*, *네트워크의 허브=게이트웨이*.

### 1-3. 의존 경계 — 헥사고날(라이트)
**코어(도메인 규칙·에이전트 로직)를 외부 세계로부터 격리**한다. 외부(DB·LLM/VLM·HTTP·스토리지)는 전부 **어댑터**로 갈아 끼우는 부품으로 취급.

```
   [ 인바운드 어댑터 ]      [ 코어 ]        [ 아웃바운드 어댑터 ]
   HTTP/SSE 라우터    →   도메인·에이전트   →   DB · LLM/VLM · 스토리지
   (외부가 코어를 부름)     (외부를 모름)      (코어가 포트로 부름)
```
- **포트** = 코어가 외부에 요구하는 인터페이스. **어댑터** = 그 인터페이스의 실제 구현.
- 라이트인 이유: `ports/`·`adapters/` 같은 폴더를 따로 만들지 않고 **친숙한 레이어 이름(라우터/서비스/리포지토리 등)이 곧 포트·어댑터** 역할을 한다. 프레임워크 형식주의는 안 함.
- 효과: **LLM·스토리지 교체 = 아웃바운드 어댑터만 교체**, 코어는 손 안 댐.

### 1-4. 도메인 — DDD(부분 차용)

**DDD(도메인 주도 설계)란?**
코드를 기술(테이블·API) 기준이 아니라 **업무(도메인) 기준**으로 묶는 접근이다. "광고", "시뮬레이션", "리포트"처럼 **현업이 쓰는 개념 단위로 코드를 나누고**, 그 개념의 규칙을 그 안에 둔다. 목적은 *비즈니스 의미가 코드 구조에 그대로 드러나게* 하는 것.

**애그리거트(Aggregate)란?**
"**항상 함께 바뀌고, 함께 일관성을 지켜야 하는 객체들의 묶음**"이다. 이 묶음에는 **대표(애그리거트 루트)** 가 하나 있고, **외부는 오직 루트를 통해서만** 내부에 접근·변경한다. 내부 객체를 외부가 직접 건드리는 걸 막아 **일관성 경계**를 만든다.

> 비유: 주문(루트)과 그 주문 항목들. 항목 수량을 바꾸려면 *주문을 통해* 바꾼다 — 그래야 "총액" 같은 규칙이 깨지지 않는다. 항목만 몰래 고치는 건 ❌.

**우리 프로젝트의 예 — `Simulation` 애그리거트**
```
Simulation  ← 애그리거트 루트 (대표)
 ├─ persona_responses  (하위)
 └─ report             (하위)

규칙: 페르소나 응답·리포트는 Simulation을 통해서만 생성/변경한다.
      외부 코드가 하위만 따로 집어 바꾸는 것 ❌ → 시뮬레이션 결과의 일관성 보장.
```

**우리는 어디까지 쓰나 (부분 차용)**
- ✅ 채택: ① 도메인별 패키징(개념 단위로 코드 분리) · ② Repository(저장소를 추상 인터페이스로) · ③ **애그리거트 경계 규율 1개**(위 `Simulation` 규칙).
- ❌ 미채택: 도메인이벤트, 애그리거트를 본격 객체로 모델링, Application/Domain 레이어 풀 분리 → **현 규모엔 과설계**. ③은 "객체"가 아니라 **지켜야 할 규칙(멘탈 모델)** 로만 차용.

### 1-5. 객체 — SOLID
별도 프레임워크가 아니라 위 구조 위에서 자연히 따라온다: **DIP**=포트로 의존 역전, **ISP**=어댑터별 좁은 인터페이스, **SRP**=도메인/레이어 모듈 분리.

---

### 왜 헥사고날-라이트인가 (클린·풀DDD를 안 쓴 이유)
- 이 프로젝트의 **변동성은 경계(어댑터)에 있다**: LLM/VLM 교체, 스토리지(local↔외부), 모달리티 라우팅 → 경계 격리를 핵심으로 하는 **헥사고날이 정확히 최적**.
- 도메인 로직은 두껍지 않다(파이프라인+CRUD 중심) → 클린의 전용 엔티티/유스케이스 레이어, 풀 DDD의 애그리거트 객체·도메인이벤트는 **지금 도입하면 비용만 큼**.
- **결론**: 헥사고날-라이트를 척추로 삼고, 클린에서는 "의존성은 안쪽을 향한다"는 **규칙 한 줄만** 빌린다. SOLID는 그 위에서 표현된다.

### 의존 방향 (불변 — 폴더명 바뀌어도 고정)
```
인바운드(라우터/HTTP·SSE) → 애플리케이션(서비스) → 아웃바운드(리포지토리) → DB
서비스 → 커널 레지스트리 → 에이전트
```
- 의존은 **항상 안쪽(코어)으로**. 역방향 ❌. (코어 격리·어댑터 교체 효과 → §1-3, 에이전트 직접 호출 금지 → §1-2)

### 결합점 = 계약(contracts) SSOT
- 모듈/트랙 간 결합은 **계약·레지스트리·SSE로만**. 직접 import 금지.
- 새 API/에이전트는 **계약부터** 정의 → 그다음 구현. (→ 병렬 작업 충돌 0)

## 2. 설정·소유 규칙

> **설정은 그것을 쓰는 레이어가 소유한다.** 어디 두느냐가 아니라 **누가 소유하는가**가 규칙.

| 관심사 | 소유 |
|---|---|
| 백엔드 런타임·의존성 (**uv** — `pyproject.toml`·lock 등) | 백엔드 (전용, 침범 ❌) |
| 프론트 런타임·의존성 (**pnpm** — `package.json`·lock 등) | 프론트 (전용) |
| DB 스키마·마이그레이션 | **DB (독립 관리)** — 변경은 항상 마이그레이션으로 |
| 시크릿(DB·인증·LLM 키) | **백엔드만** — 프론트가 키 필요 시 백엔드가 프록시 |
| 프론트 환경변수 | 프론트 — **공개값만**(번들 노출), 시크릿 ❌ |
| 배포 오케스트레이션 | 공용(레이어 밖) — 특정 레이어 안에 두지 않음 |

→ **uv 관련 설정은 백엔드에, pnpm 관련 설정은 프론트에** 둔다. 각 레이어의 의존성 도구는 그 레이어가 단독 소유한다.

→ **기술 스택에 필요한 라이브러리는 매니페스트에 선언**한다: 백엔드는 `pyproject.toml`, 프론트는 `package.json`. 개발에 쓴 라이브러리를 코드만 추가하고 매니페스트에 안 적으면 ❌ — 다른 작업자 환경에서 재현 안 됨. **"설치 = 매니페스트에 명시 + lock 갱신"** 을 한 묶음으로.

`.env`는 쓰는 레이어가 소유(백엔드용/프론트용 분리). 두 레이어 가로지르는 단일 `.env` ❌.

## 3. 배포 토폴로지 (단일 EC2)

> 프론트 + 백엔드를 **하나의 EC2**에 함께 올린다. (FE=Vercel 분리 ❌ — 변경됨)

- **엣지 게이트웨이(리버스 프록시)는 하나** — 외부 진입점·TLS 종단·경로 분기(프론트 vs `/api`+SSE)를 한 곳이 책임. 전용 API Gateway 제품 ❌(과설계).
- **앱 내부 라우터는 도메인별 N개**(합치지 않음 — §1 경계). 단 단일 앱·단일 `/api` 표면 아래 마운트 → 노출 표면은 하나.
- 프론트·백엔드·프록시를 컨테이너로 **한 번에** 띄움 → 프론트도 **컨테이너화 대상**.
- DB·오브젝트 스토리지는 **외부(관리형)** 유지 — EC2에 얹지 않음.
- **전제 묶음**: 엣지 하나 = 백엔드 **단일 인스턴스** 전제(현재 `--workers 1` + 인메모리 SSE 버스). 수평 확장 시 SSE 버스 외부화 필요.
- 단일 호스트라도 **§1·§2 경계 규칙은 그대로** — 한 박스에 올라간다고 결합하지 않는다.
